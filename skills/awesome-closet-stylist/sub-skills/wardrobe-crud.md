# wardrobe-crud

衣橱增删改查规约。处理所有与衣物数据变更或查询相关的请求。

---

## 数据 Schema

### 必填字段（最小可用记录）

写入衣橱前必须具备以下三个字段，缺任一项则不执行写入，先追问：

| 字段 | 类型 | 描述 | 示例 |
|---|---|---|---|
| `id` | string | 稳定唯一标识，不因更名而变化；格式：`{类别缩写}-{两位序号}` | `top-01`、`out-03` |
| `category` | string（枚举） | 衣物类别 | `outerwear` \| `top` \| `bottom` \| `dress` \| `shoes` \| `bag` \| `accessory` |
| `name` | string | 用户能认出来的自然语言名称 | `棕色宽松西装外套` |

**`id` 生成规则**：检查现有 id 列表，取同类别最大序号 +1，避免冲突。

### 可选字段

以下字段按需录入，信息已足够区分衣物时直接写入最小记录，不强制采集：

| 字段 | 类型 | 描述 | 取值范围 / 说明 |
|---|---|---|---|
| `sub-category` | string | `category` 的子分类，自由文本，但尽量归纳至已有标签，避免为同类物品造不同词（如：T恤统一写 `t-shirt`，休闲裤统一写 `chinos`） | `t-shirt`、`hoodie`、`chinos`、`sneakers`、`tote` 等 |
| `color` | string | 主色，自然语言描述 | 自由文本，如 `棕色`、`橄榄绿`、`褪色黑`、`黑白色` |
| `seasons` | string[] | 适穿季节数组 | 枚举：`"spring"` · `"summer"` · `"autumn"` · `"winter"`；中文季节描述可推断映射 |
| `style` | string | 风格标签 | 自由文本，如 `宽松/西装`、`夹克`、`针织/开衫`；多个风格用 `/` 分隔 |
| `material` | string | 面料描述 | 自由文本，如 `棉`、`毛呢`、`牛仔` |
| `warmth` | string（枚举） | 保暖等级，辅助穿搭推荐 | 标准值：`light` · `medium` · `heavy`；允许简写 `薄` · `厚` |
| `notes` | `{ date: string; text: string }[]` | 带日期的备注日志，只做增量追加，格式见下方说明 | `[{ "date": "2026-04-04", "text": "难以搭配" }]` |
| `image_refs` | string[] | 图片路径或引用标识数组 | 文件相对路径或资产 ID |

### 字段约束说明

- `id`：只增不改，写入后不因衣物信息变更而修改
- `seasons`：使用英文枚举，推断规则：春→`spring`、夏→`summer`、秋→`autumn`、冬→`winter`
- `warmth`：不强制录入；录入时优先标准枚举，`薄` = `light`，`厚` = `heavy`
- `notes`：数组格式，每条记录包含 `date`（`YYYY-MM-DD`）和 `text`；**只允许追加新条目，不得修改或删除历史条目**；若用户明确推翻旧备注，追加一条新的说明性条目（如 `{ "date": "2026-04-04", "text": "之前备注已过时，实际穿搭频率较高" }`），而非直接覆盖

## 新增衣物

1. 一旦意图清晰，立即写入 `user/wardrobe.json`，不等待所有字段齐全。
2. 生成 `id` 时，检查现有 id 列表，取同类别最大序号 +1，避免冲突；格式：`{类别缩写}-{两位序号}`，如 `top-03`。
3. 只有在无法确定 `category` 或 `name` 时，才追问一个问题。
4. 写入后，用自然语言简短确认已记录，不逐字段复述。

## 修改衣物

1. 先在衣橱中定位目标衣物。
2. 如果用户描述可以唯一匹配到一件衣物，直接修改，无需确认。
3. 如果描述模糊，匹配到多件候选时，列出候选并询问"你说的是哪一件？"。
4. 修改后用自然语言简短确认变更内容。

## 删除衣物

1. 删除前必须能唯一确定目标衣物，否则先澄清。
2. 删除是不可逆操作，如果用户语气像是顺带提及（非明确指令），确认一次再执行。
3. 删除后简短确认。

## 查询 / 筛选

- 只读，不修改数据
- 先给一句自然语言总结（"你有 X 件适合春天的外套"），再列出最相关的衣物
- 按用户关心的维度筛选（季节、类别、颜色、风格），不要像数据库导出一样逐字段堆砌
- 如果查询结果为空，明确告知，并可提示用户当前衣橱中同类别的情况

---

## 追问策略

只有在以下情况才允许追问：
- 无法确定新增衣物的 `category` 或 `name`
- 修改/删除目标模糊，无法安全执行
- 用户的查询条件无法从现有数据中解析

每次只问一个最关键的问题。
