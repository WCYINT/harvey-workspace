# user-preference

用户偏好异步维护规约。负责识别对话中的偏好信号，并静默更新 `user/preference.json`。

---

## 偏好数据文件

路径：`user/preference.json`

Schema 示例：

```json
{
  "style_affinities": ["简约", "通勤"],
  "style_aversions": ["过于花哨", "荧光色"],
  "color_preferences": {
    "preferred": ["深灰", "藏蓝", "白"],
    "avoided": ["大红", "荧光黄"]
  },
  "occasion_habits": {
    "work": "偏正式但不过度，常穿西装外套或毛呢大衣",
    "weekend": "轻松休闲，偏好运动风或简约 casual"
  },
  "fit_preferences": "偏好宽松版型，不喜欢过于修身",
  "misc_notes": []
}
```

首次读取若文件不存在或为空对象，视为"偏好未初始化"，按正常流程推荐，偏好信号累积后逐步写入。

---

## 偏好信号识别

以下类型的用户表达应触发偏好更新：

| 信号类型 | 示例表达 |
|---|---|
| 风格正向 | "我喜欢这种简约的感觉"、"这套挺符合我的风格的" |
| 风格负向 | "我不太喜欢太花的东西"、"这件颜色太跳了" |
| 颜色偏好 | "我最近很喜欢穿深色系"、"我不太驾驭红色" |
| 版型偏好 | "我比较喜欢宽松一点的"、"这个版型太修身了不适合我" |
| 场合习惯 | "我上班很少穿这么休闲的"、"我平时周末才穿运动风" |
| 通用标注 | "帮我记住我不太穿裙子"、"记一下我不喜欢高领" |

---

## 更新行为

1. **静默写入**：识别到偏好信号后，在生成主回复的同时更新 `user/preference.json`，不中断对话、不另起一轮告知用户。
2. **渐进累积**：偏好数据持续叠加，不覆盖已有的有效偏好，除非用户明确纠正之前的偏好。
3. **冲突处理**：如果新信号与已有偏好冲突（例如之前记"喜欢蓝色"，现在说"不太喜欢蓝色"），以更新的信号为准，并更新对应字段。
4. **不做确认**：不需要每次更新都问用户"我帮你记下来了，对吗？"，除非用户明确要求确认或变更内容对推荐影响很大。

---

## 偏好在推荐中的使用

`outfit-recommendation.md` 在推荐时应读取 `user/preference.json`，将偏好作为调权因子：

- `style_aversions` / `color_preferences.avoided`：从候选中降权或排除
- `style_affinities` / `color_preferences.preferred`：优先选择
- `fit_preferences`：在同类单品中优先匹配版型描述吻合的
- `occasion_habits`：结合场合时参考用户历史习惯

---

## 追问策略

不主动追问偏好。偏好数据通过自然对话渐进积累。
只有用户明确说"帮我更新一下我的偏好"或类似指令时，才可以主动询问具体变更内容。

---

## Hooks 自动整理

当本次 session 首次识别到偏好信号时，按 `SKILL.md` 中"Hooks 自动整理"章节的判定流程处理。

判定与提示在 session 级别只执行一次，由首个触发的 sub-skill 负责；若另一个 sub-skill 后续也触发，不再重复提示。
